## Question 3

**Question:** The `nerdtui-spec.md` specifies that the SQLite store persists `hashrate_history` for rendering the 60-second sparkline across restarts. Do the cumulative variables `SharesFound` and `BestDifficulty` also need to survive restarts (requiring a separate key-value/config table in SQLite to persist them), or are they strictly transient in-memory values that reset to `0` when the program terminates?
**Impact:** If persistence is required, we must update the SQLite schema in `internal/store/migrations.go` to include a table for cumulative statistics (e.g., `miner_stats`), and add store methods to load and update these values.
**Status:** ✅ Respondida

**Response:** `SharesFound` and `BestDifficulty` are strictly transient in v1. They reset to zero on every restart. Only `hashrate_history` is persisted. Persisting cumulative counters without a session timestamp would produce an ambiguous number the user cannot interpret (is that 42 shares today or across 6 months?). The SQLite schema remains with a single `hashrate_history` table. Per-session statistics (shares, best difficulty, start/end time) are a v2 feature requiring a `miner_sessions` table with temporal context. This constraint must be captured as `REQ-STORE-SCOPE-01` in `internal/store/requirements.md`.

**Impact:** If network submission is required, `StratumPoolClient` must support composing and writing `mining.submit` JSON-RPC messages over the TCP connection and handling response errors. If local-only, we skip the network write logic entirely when a share is found.
**Status:** ✅ Respondida

**Response:** `mining.submit` must be sent. The non-goal "real share submission" refers to the expected outcome (winning a block reward), not to omitting the protocol message. Skipping `mining.submit` would cause the pool to drop the connection, breaking the job feed and making `SharesFound` a meaningless local counter. When `MinerWorker` emits `ShareFoundMsg`, `StratumPoolClient` must call `SubmitShare(ctx, share) (bool, error)`, send the `mining.submit` JSON-RPC over the existing TCP connection, and await the pool response with a 10-second timeout. The boolean result (`accepted`/`rejected`) is included in `ShareFoundMsg.Accepted` and displayed on Screen 1. A submission failure (timeout or broken connection) emits `PoolErrorMsg` but does not interrupt the mining loop. In `MockMining=true` mode, `MockPoolClient.SubmitShare` returns `true` unconditionally without network I/O. `SubmitShare` must be added to the `PoolClient` interface. This must be captured as `REQ-WORKER-SUBMIT-01` in `internal/worker/requirements.md`.
